Personal Site TUI Part 1

  • go
  • devops

In my spare time I have been building a Claude wrapper. It’s on around its 4th or 5th iteration by now. I have gone from pure text files, to more pure text files, to a whole Electron app (this one took a long time), and then I decided to make a Terminal User Interface (TUI). First I went Rust because I was looking for something new to learn, then TypeScript to make it work better with the Agent SDK (TypeScript or Python at the moment), and then when I was fighting against Ink — which is what Claude Code is currently written in, as far as I know — I found Charm. The wrapper is still ongoing, but I wanted a smaller project I could do in public and explore some parts of tech that I don’t know. All I needed was an idea.

Be my guest

I thought a little guestbook would be nice — someone SSHs into my site, gets a terminal interface, and can leave a message that shows up on the web. So the plan is a Go backend with a TUI, a database to store the entries, and an HTTP API to return them to the frontend. One binary, two interfaces.

A Drop in the Ocean

The first step is picking a cloud provider. I didn’t want something overly complex, but I also didn’t want something completely “plug-and-play”. I don’t have a lot of experience with devops, so dealing with a real VM felt like the right level of friction. Because of this I chose DigitalOcean.

A basic droplet — 512MB RAM, 1 CPU, 10GB SSD — costs $4 a month. Enough to run a Go binary and a SQLite file. I skipped the managed database upsell ($15/month for infrastructure I’d never use) and picked Ubuntu 24.04.

A Guided Dip

I am using Claude to guide me through this. The thing I’ve come to appreciate about working with AI in unfamiliar territory is that it gives you a knowledgeable companion without the awkwardness of asking a person obvious questions. I asked a lot of obvious questions.

It turned out I already had an SSH key on my machine (id_ed25519 — better than the older RSA format, apparently). I pasted the public key into DigitalOcean during setup and connected straight in as root. First thing Claude had me do was create a normal user and stop logging in as root, which I did. The user creation process on Ubuntu asks for your room number, which I thought was funny.

After that: disable root login in the SSH config, restart the SSH daemon, and set up ufw — the firewall. Allow SSH, enable it, done. The whole thing took about fifteen minutes and now I have a locked-down server sitting in a data centre somewhere.

Dependencies

With the server secured, we need two things installed: Go and nginx.

Go is straightforward — sudo apt install golang-go — and we end up with version 1.22.2. nginx takes one command too. What nginx does is sit in front of the Go application and act as a reverse proxy. A reverse proxy is a confusing name for a simple idea: it accepts requests from the internet on behalf of your app, handles things like TLS, and forwards traffic through. Your app never gets exposed directly. We open the firewall for it and move on.

A Record

The last thing for today is pointing a subdomain at the droplet. I head into Namecheap, add an A record with the host set to tui and the value set to the droplet’s IP, and now tui.matthewwbuckley.com exists in the world. DNS propagation can take a while, but it comes through quickly.

HTTPS

With the subdomain live, we can get a certificate. We install certbot and run it against the domain. Let’s Encrypt proves you own the domain by temporarily serving a file at a known URL and checking it’s reachable — if it is, you clearly control the server behind that domain. Certificate issued. certbot also edits the nginx config to use it and sets up a cron job to auto-renew every 90 days. The whole thing is automated and free, which still feels slightly miraculous.

nginx config

The last piece is telling nginx what to actually do with incoming requests. By default it just serves a placeholder page. We create a config file in /etc/nginx/sites-available/ — the filename is just a label for your own sanity, it means nothing to nginx. What matters is the server_name line inside it, which is what nginx actually uses to route traffic to the right place.

The config says: any request for tui.matthewwbuckley.com, forward it to localhost:8080. That’s where the Go app will listen. There’s nothing on port 8080 yet, but the plumbing is in place.

One thing I keep noticing in this kind of work: some things are meaningful and some things are just labels, and nothing tells you which is which. You have to ask, or find out the hard way.

The server is ready. Next time I write some Go, which should be interesting because I don’t know any Go.
