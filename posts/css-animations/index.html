Kinetic CSS

  • css
  • design
  • code

You don’t need JavaScript for movement. A few keyframes, some pseudo-elements, and suddenly a dot traces a square endlessly. Hypnotic.

@keyframes trace {
  0%,
  100% {
    transform: translate(0, 0);
  }
  25% {
    transform: translateX(40px);
  }
  50% {
    transform: translate(40px, 40px);
  }
  75% {
    transform: translateY(40px);
  }
}

The web has had animation capabilities for over a decade now. CSS transitions landed in 2009. Keyframe animations followed shortly after. Yet we still reach for JavaScript libraries to make things move.

There’s a purity to CSS animation. It runs on the compositor thread, completely divorced from the main thread where JavaScript lives. Sixty frames per second, buttery smooth, even on a struggling mobile chip.

The trick is thinking in states rather than instructions. You don’t tell the browser “move right, then down, then left, then up.” You define waypoints—where should this element be at 25%? At 50%?—and the browser interpolates the rest.

Pseudo-elements are the other secret weapon. A single div can spawn two more elements via ::before and ::after. Three moving parts from one HTML node. No wrapper divs, no JavaScript state management.

I’ve been building little kinetic sculptures this way. Dots orbiting dots. Lines extending and retracting. Squares rotating inside squares. Each one is just a handful of CSS rules, yet they feel alive.
